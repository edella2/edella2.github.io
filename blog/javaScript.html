Ruby Hashes vs. JavaScript: Objects

Round 1 - FIGHT!

10/25/2015

Hearing "hashes" when using Ruby used to send shivers up my spine. I was horrible with them, they were confusing to me and I just never seemed to be able to use them. I never thought I would have to deal with something I thought was so complicated and confusing again until this week when I met... JavaScript: Objects.

If you read my other blog about arrays vs hashes a few weeks ago, you would see that despite having trouble learning to use hashes I had to concede that they were super useful.

To define a hash in ruby we would type something like the following:

eric = { name: "Eric", age: 28, job: "Student" home: "SF, CA" }

To define a object in JavaScript we would have to type:

var eric = { name: "Eric", age: 28, job: "Student" home: "SF, CA" }

... Wait that's pretty much the same thing. The differences between JavaScript and Ruby lie between how the keys and key values are stored. In ruby the key ( i.e. name, age) can be anything from strings, integers or symbols. While in JavaScript the keys have to be strings.

But don't rule JavaScript out yet for only storing key values as strings. It has a few other cool things you might not know about. Objects and their key values can be accessed using dot notation. If I wanted to access my age in JavaScript I could type:

eric.name

or

eric["name"]

Having multiple ways to access information stored in an object is a great thing for flexibility (and readability!)

Other differences between the two exist with how we can interact with the information stored in a hash or object.

Ruby has several built in class methods and enumerables, such as hash_name.each that interact with hashes that makes iterating through a hash a breeze.

These same enumerables do not exist with JavaScript because Objects also work in a similar fashion as classes in Ruby. Objects in JavaScript can store methods as well inside the hash!(a further in depth explanation on that subject for round 2)
